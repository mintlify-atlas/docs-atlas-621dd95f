## Prerequisites

- **Node.js 18+** installed on your machine
- The backend API running at `http://localhost:3001` before you start the frontend

<Note>
  The frontend depends on the backend for product listings, payment processing, and transaction status polling. Start the backend first — see the [backend installation guide](/backend/installation).
</Note>

## Setup

<Steps>
  <Step title="Navigate to the frontend directory">
    ```bash
    cd frontend
    ```
  </Step>

  <Step title="Install dependencies">
    ```bash
    npm install
    ```
  </Step>

  <Step title="Configure environment variables">
    Copy the example file and fill in your Wompi public key:

    ```bash
    cp .env.example .env
    ```

    Open `.env` and set `VITE_PAYMENT_PUBLIC_KEY` to your Wompi sandbox public key. The other two variables have working defaults for local development:

    ```env
    VITE_API_URL=http://localhost:3001/api
    VITE_PAYMENT_API_URL=https://api-sandbox.co.uat.wompi.dev/v1
    VITE_PAYMENT_PUBLIC_KEY=pub_stagtest_xxxxxxxxxxxx
    ```

    See [Configuration](/frontend/configuration) for a full description of each variable.
  </Step>

  <Step title="Start the development server">
    ```bash
    npm run dev
    ```

    The app is now available at **http://localhost:5173** with Hot Module Replacement enabled.
  </Step>
</Steps>

## Available commands

| Command | Description |
|---------|-------------|
| `npm run dev` | Start the development server at `http://localhost:5173` with HMR |
| `npm run build` | Compile TypeScript and build the production bundle |
| `npm run preview` | Serve the production build locally for inspection |
| `npm run lint` | Run ESLint across the source tree |
| `npm test` | Run the test suite once with Vitest |
| `npm run test:watch` | Run tests in watch mode |
| `npm run test:coverage` | Run tests and generate a coverage report |

<Tip>
  `npm run build` runs `tsc -b` before Vite bundles the output, so TypeScript type errors will cause the build to fail. Use this as a quick type-check in CI or before deploying.
</Tip>
